# 赛博分身协同项目 · 项目概要

> 版本：v0.1（理解对齐稿）
> 更新日期：2026-04-26

---

## 一、项目愿景

**核心命题**：AI 不是人类职业的"终结者"，而是组织沟通熵增的"过滤器"。

我们不利用 AI 裁撤员工，而是利用 AI 重塑协同：
为产品、研发、美术等不同职能构建具备专业逻辑的 **赛博分身（Digital Twin）**，
让分身在虚拟空间进行高频的需求对齐与冲突检测。

当原本需要十人开会、反复拉齐的"微小字段修改"转变为分身之间秒级的自动化逻辑校验时，
我们消灭了琐碎的沟通内耗，让真实的人类从"工具人"的同步等待中解放出来，
回归到 **定义美学、驱动创新、做高阶判断** 的战场。

**模式口号**：**分身协同，人类裁决。**

---

## 二、核心机制

| 层级 | 角色 | 职责 |
|------|------|------|
| **分身层** | 每个员工的 Digital Twin | 掌握本人的职能逻辑、工作上下文、历史决策；代理日常协同 |
| **协同层** | 分身之间的 A2A 通讯协议 | 需求对齐、字段确认、接口校验、冲突检测，秒级完成 |
| **知识层** | 企业知识库 + 带权重的 RAG | 让分身不必反复"问人"也能拿到事实 |
| **裁决层** | 人类本体 | 只在分身协同的产物上做最终确认/决策；不确定的事项主动 escalate 给本人 |

---

## 三、分身能力模型

分身是 **"混合型代理 + 主动 escalation"** 的形态，具备三类能力：

### 3.1 决策与沟通（轻量代理）
- 代替本人回复常规消息（状态同步、信息确认、进度问询）
- 在已授权范围内做轻量决策（例如确认非关键字段、批准小额事项）
- 在群聊/私聊中以本人身份或"分身"身份发言（需明确标识策略）

### 3.2 工具执行（轻量任务）
- 拉报表、查数据、汇总文档
- 调用飞书的多维表格、文档、审批等 API
- 执行预定义的标准化操作（建会、发通知、催办）

### 3.3 主动升级（Escalation）
> 这是分身可信度的关键机制

当分身遇到以下情形时，**主动推送本人接管**：
- 超出已授权范围的决策（例如金额、人事、产品方向）
- 知识库无法覆盖的事实空缺
- 多个分身协同后产生冲突且无法自动调和
- 检测到自身置信度低于阈值

升级方式：飞书消息推送 → 附带上下文摘要 + 建议选项 → 人类一键决策回流。

## 四、知识层设计

### 4.1 聊天记录 → 知识库的闭环

```
飞书聊天 (群/私聊)
      │ 定期抓取
      ▼
   Compact 处理        ← 提炼事实、决策、待办、人物关系
      │
      ▼
  企业知识库入库
      │
      ▼
   分身 RAG 召回
```

**意义**：把散落在 IM 中的隐性知识显性化。
**类比**：Claude 的 auto memory 机制，但作用在组织维度。

### 4.2 带权重的 RAG（Cache 化设计）

知识条目带 `weight`，受两个维度影响：

- **Recency（时效性）**：越近期的知识权重越高，按时间衰减
- **Frequency（命中频次）**：越常被检索/命中的知识权重越高（LFU 思路）

召回时优先命中高权重条目，等价于在 RAG 之上加了一层 **hot cache**：
- 提升召回相关性（近期/常用 > 陈旧/冷僻）
- 降低召回延迟（热数据常驻，冷数据下沉）
- 实现机制类似 LRU/LFU + 向量检索的混合层

## 五、技术栈

- **平台基座**：飞书（Lark）开放平台
- **核心 SDK**：飞书 CLI / Open API
- **I/O 通道**：飞书消息、群、文档、日历、审批、多维表格——分身的"手脚"都在飞书生态内
- **Agent 框架**：Claude（自带 memory，适合做分身的人格/上下文持久化）
- **知识库**：向量库 + 权重层（待选型）

## 七、MVP 场景：产品改字段的协同闭环

**场景描述**：
产品经理需要修改一个产品字段（例如"用户等级"的取值范围由 1–5 改为 1–10）。
传统流程：拉群 → @ 研发评估 → @ 设计同步 → @ 测试同步 → 反复对齐 → 一周。

**分身协同流程（目标）**：

```
1. 产品本人 → 告诉自己的分身："把用户等级改成 1–10"
              │
              ▼
2. 产品分身  → 解析变更 → A2A 通知研发分身、设计分身
              │
              ▼
3. 研发分身  → RAG 检索影响范围 → 评估改动成本 → 回报
   设计分身  → RAG 检索关联设计 → 评估视觉影响 → 回报
              │
              ▼
4. 产品分身  → 汇总各方反馈 → 生成结论摘要
              │
              ├─ 若分身可达成一致 → 推送产品本人确认（一键 OK）
              └─ 若存在冲突 / 不确定 → escalation：推送相关本人接管
              │
              ▼
5. 人类本体  → 一键裁决 → 分身回写决议到飞书文档/多维表格
```

**为什么选这个场景**：
- 高频、低决策风险，适合分身代劳
- 涉及多角色协同，能充分验证 A2A 通讯
- 结果可量化（节省的会议数 / 拉群次数）

## 八、关键设计原则

1. **人类是裁决者，不是执行者**：分身做事，人做判断。
2. **可解释性优先**：分身的每个动作/回复都要可追溯（谁的分身、依据什么知识、为什么这么做）。
3. **保守的授权边界**：默认收紧，宁可多 escalate，也不擅自决策。
4. **沟通可旁路**：人类可随时介入分身对话，接管或纠正。
5. **知识沉淀闭环**：每次协同的产物（决议、冲突、纠正）都回流到知识库，让分身越用越准。

## 九、待对齐 / 待决策问题

> 这些是当前还没敲定的关键点，影响后续设计

1. **分身的身份呈现**：在飞书里以"本人头像"还是"分身标识"发言？涉及信任和透明度。
2. **授权粒度**：怎么定义"轻量决策"的边界？是否需要每个员工自定义授权范围？
3. **隐私边界**：聊天记录抓取的范围？私聊是否纳入？需要怎样的合规设计？
4. **分身人格化**：分身是否模仿本人语气？还是统一中性风格？
5. **冷启动**：新员工的分身如何初始化（缺少历史数据）？
6. **多分身冲突仲裁**：分身之间僵持时的仲裁机制（轮询人类？还是有专门的"调解分身"？）
7. **知识库选型**：自建还是基于飞书 Wiki + 向量层？
