## 三、决策详情

### D-001 知识库走自建向量层

**状态**：已决
**日期**：2026-04-28

**背景**：
项目概要 §九.7 提出"自建 vs 飞书 Wiki + 向量层"的悬而未决问题。这个选型决定后续整个知识层的实现路径，是技术选型的前置依赖。

**选项**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **自建向量层** ✅ | 加权 RAG（recency × frequency）可深度定制；数据所有权清晰；不被平台接口限制 | 需自己维护向量库、ETL、检索服务 |
| 飞书 Wiki + 飞书向量层 | 接入快、与飞书生态原生整合 | 加权策略受限；飞书向量能力开放程度未知；定制空间小 |
| 纯云向量服务（Pinecone 等） | 免运维 | 数据出域，企业场景敏感；成本不可控 |

**决策**：选自建向量层。

**理由**：
- 项目核心特色是"带权重的 RAG"，需要对召回打分逻辑有完全控制权
- 比赛阶段需要快速迭代召回策略（衰减系数、命中频次窗口等），托管方案改起来不灵活
- 数据闭环（聊天 → compact → 入库 → 召回）涉及多次写入和元数据更新，自建可避免接口限制

**后果与权衡**：
- 多了向量库运维成本（D-005 选 pgvector 一定程度上缓解，可与业务库共用 PG 实例）
- 需自己实现 ETL pipeline 和加权层（在 MVP 骨架中体现）

### D-002 分身身份采用"本人头像 + 分身标识"

**状态**：已决
**日期**：2026-04-28

**背景**：
项目概要 §九.1。分身在飞书中发言的可见身份直接影响信任、合规与后续接入方式。

**选项**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **本人头像 + 分身标识**（如名字后挂"·分身"） ✅ | 保留个人辨识度；明确告知"这是分身在说话"，不欺瞒 | 需在每条发言中维持标识一致性 |
| 纯本人头像 | 沟通最自然 | 模糊真人/AI 边界，存在伦理与合规风险 |
| 完全独立的分身账号 | 边界最清晰 | 失去"分身代表本人"的语义；账号管理成本高 |

**决策**：本人头像 + 分身标识。

**理由**：
- 与项目原则 §八.2"可解释性优先"一致：每条消息都能被识别为"分身行为"
- 满足项目原则 §八.4"沟通可旁路"：人类介入时身份切换有清晰视觉差异
- 比"独立账号"方案接入飞书所需权限更轻

**后果与权衡**：
- 实现层面需规范化"分身标识"的展示形式（昵称后缀 / 卡片 footer / 消息开头标记），具体形态在 MVP 骨架中确定
- 需进一步研究飞书的接入路径：用"代发消息"还是"以应用身份发消息但带 @本人"——留到技术选型文档展开

### D-003 聊天记录抓取默认仅群聊，私聊需授权

**状态**：已决
**日期**：2026-04-28

**背景**：
项目概要 §九.3。聊天数据是知识库的主要来源，但隐私边界在企业场景中是合规红线。

**选项**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **默认仅群聊，私聊 opt-in** ✅ | 平衡知识覆盖与隐私；群聊本就是协同语境，符合用户预期 | 私聊中的关键决策可能漏采 |
| 全部抓取 | 知识覆盖最全 | 隐私风险大、合规阻力大、用户抵触 |
| 全部不抓（仅文档/Wiki） | 零隐私争议 | 知识来源贫乏，违背项目核心闭环设计 |

**决策**：默认仅群聊；私聊默认不抓，但允许用户主动授权后纳入。

**理由**：
- 群聊本质是多人协同记录，参与者预期"会被看见"，隐私期望低
- 私聊往往涉及个人/敏感信息，强制抓取会引发抵制
- opt-in 机制把控制权交给用户，符合 §八.3"保守的授权边界"

**后果与权衡**：
- 知识层会有覆盖盲区（私聊里发生的关键对齐），需要在 escalation 与冲突检测时有兜底（让分身意识到"知识不足"并 escalate）
- 需要设计授权 UI 与撤回机制（私聊抓取的开关，可随时关停并清除已采数据）
- 私聊抓取范围还需进一步定义（"全部私聊"还是"指定对象"），见 P-005

### D-005 向量库采用 pgvector

**状态**：已决
**日期**：2026-04-28

**背景**：
D-001 决定自建后，需要选具体的向量库。

**选项**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **pgvector** ✅ | 单 PG 实例存向量 + 元数据 + 业务表；加权 SQL 写起来直观；运维成本低 | 极大规模（千万级以上）召回性能弱于专用库 |
| Qdrant | 性能好、过滤能力强；docker 一键起 | 多一个独立服务；元数据/业务库分两处 |
| Milvus | 性能最强 | 部署重、集群运维门槛高，对 MVP 过载 |
| Chroma | 嵌入式、最轻量 | 多进程并发与持久化能力弱 |

**决策**：pgvector。

**理由**：
- 加权 RAG 需要把 `weight = f(recency, frequency)` 与向量召回结合，pgvector 用一条 SQL（`ORDER BY (embedding <=> :q) / weight`）即可表达
- frequency 计数本质是事务性更新，单库一致性比跨库简单得多
- MVP 数据规模远不到 pgvector 的瓶颈（百万级以下完全胜任）

**后果与权衡**：
- 数据规模膨胀后可能要迁移到 Qdrant/Milvus，但 schema 抽象做好后迁移代价可控
- PG 需启用 pgvector 扩展，部署文档要写清楚

### D-007 A2A 传输采用进程内 asyncio 事件总线

**状态**：已决
**日期**：2026-04-28
**关联**：[技术选型.md §3.3](./技术选型.md)

**背景**：
分身之间需要通讯（A2A），传输层直接影响进程模型与运维复杂度。

**选项**：

| 方案 | 优点 | 缺点 |
|---|---|---|
| **进程内 asyncio 事件总线** ✅ | 零基础设施；端到端单机可跑通；调试简单 | 不能跨机扩展 |
| Redis pub/sub | 跨进程；解耦 | 多一个依赖 |
| 飞书消息当总线 | 语义直观 | 增加延迟；受 rate limit |
| HTTP/gRPC 互调 | 边界清晰 | 对 MVP 过重 |

**决策**：进程内事件总线，对外抽象成 `MessageBus` 接口。

**理由**：MVP 单机部署足够；抽象接口为未来替换 Redis/NATS 留余地。

**后果与权衡**：分身间通讯逻辑必须经过 `MessageBus` 接口（不允许直接函数调用），否则后续替换会暴露耦合点。

### D-012 加权公式：`cosine × exp(-λ·age) × (1+log(1+hits))`，λ=0.05

**状态**：已决
**日期**：2026-04-28
**关联**：[技术选型.md §3.4.3](./技术选型.md)

**背景**：
"带权重的 RAG"是项目核心特色，需要把概念落到具体公式。

**选项**：

| 维度 | 候选 |
|---|---|
| Recency 衰减 | 线性 / **指数** ✅ / 阶梯 |
| Frequency 饱和 | 直接计数 / **对数** ✅ / Sigmoid |

**决策**：
- recency：`exp(-λ × age_in_days)`，λ=0.05（半衰期约 14 天）
- frequency：`1 + log(1 + hit_count)`
- 最终：`final_score = cosine × recency × frequency`

**理由**：指数衰减平滑可调；对数饱和避免高频热点垄断召回；λ 由配置文件可调，比赛期可基于真实数据 A/B 调参。

**后果与权衡**：参数初值需在第三阶段基于真实数据微调；不同业务场景可能需要不同的 λ（暂留作后续优化）。

### D-013 A2A 超时层级：30s / 5min / 15s

**状态**：已决
**日期**：2026-04-28
**关联**：[A2A协议草案.md §七](./A2A协议草案.md)

**背景**：
A2A 通讯需要明确超时阈值，否则分身会无限等待或频繁误判。

**决策**：
- 单条消息超时（如 request 等待 response）：30 秒
- Conversation 总超时：5 分钟
- Heartbeat 间隔：15 秒（连续 3 个间隔未到视为接收方失联）

**理由**：
- 30 秒覆盖正常 LLM 推理 + RAG 检索时间，超过即视为分身实际异常
- 5 分钟是人类协同的"关注上限"，超过即应升级
- 15 秒 heartbeat 让长任务能续命，又不至于过度刷消息

**后果与权衡**：阈值由配置文件可调；LLM 推理慢链需 ack + heartbeat 显式续命。

### D-014 A2A 重试策略：超时仅自动重试 1 次

**状态**：已决
**日期**：2026-04-28
**关联**：[A2A协议草案.md §7.2](./A2A协议草案.md)

**背景**：
单条消息超时后是否自动重试、重试几次。

**选项**：

| 方案 | 优缺点 |
|---|---|
| 不重试，直接 escalate | 灵敏但偶发抖动会浪费人类介入 |
| **重试 1 次后 escalate** ✅ | 平衡偶发故障与升级时效 |
| 多次指数退避 | 网络通讯惯例，但进程内总线无意义 |

**决策**：超时后自动重试 1 次（同 conversation_id，新 message_id），仍超时则 escalate。

**理由**：进程内总线丢消息说明分身实质故障，多次重试只拖延 escalation；1 次重试覆盖偶发瞬时阻塞。

**后果与权衡**：严重故障时升级延迟 = 30s × 2 = 60 秒；重试消息要保持 payload 一致（幂等约束）。

**状态**：已决
**日期**：2026-04-28
**关联**：[A2A协议草案.md §六](./A2A协议草案.md)

**背景**：
分身能力（处理什么 intent）需被 `MessageBus` 知道才能路由 request。

**选项**：

| 方案 | 优缺点 |
|---|---|
| **静态 yaml 注册表** ✅ | 启动加载即可；运维直观 |
| 运行时 capability 声明 | 灵活；需要发现协议 + 同步机制 |
| 中心化 capability 服务 | 适合大规模；MVP 过重 |

**决策**：MVP 使用 `config/twins.yaml`，启动时加载到 `MessageBus` 路由表。

**理由**：MVP 分身数量 < 10，静态配置足够；动态发现与"先跑通"目标冲突。

**后果与权衡**：新增/修改分身需重启服务；动态发现进 P-006。

### D-016 Conflict 仲裁直接 escalate，无调解分身

**状态**：已决
**日期**：2026-04-28
**关联**：[A2A协议草案.md §九](./A2A协议草案.md)；呼应 P-003

**背景**：
两个分身意见不一致时，是否引入"调解分身"做自动仲裁。

**决策**：MVP 不引入调解分身；分身僵持时 1 轮互发 propose 重审，仍冲突则直接 escalate 给人类。

**理由**：
- 调解分身需要"权威性"或"领域优先级"判定，本身是复杂建模
- MVP 阶段冲突量小，人类裁决足够
- 与项目原则 §八.1"人类是裁决者"一致

**后果与权衡**：高冲突场景人类负担可能加重；可通过事后分析常见冲突类型，再决定是否引入 arbiter（P-008）。

### D-017 Memory 隔离：跨分身必须经 A2A 显式 request

**状态**：已决
**日期**：2026-04-28
**关联**：[A2A协议草案.md §10.3](./A2A协议草案.md)

**背景**：
单进程内多个分身技术上能直接读对方对象状态，需明确边界。

**决策**：
- 分身 A 不允许直接访问分身 B 的 memory 或私有状态
- A 需要 B 的信息时必须发 `type=request` 的 A2A 消息
- B 在 `response` 中决定暴露什么（基于自身 policy）

**理由**：
- 可解释性：所有跨分身的信息流动都进 `a2a_messages` 表
- 隐私边界：B 的 owner 通过 policy 控制暴露范围
- 跨语言/跨进程兼容：未来分身拆服务时无需改逻辑

**后果与权衡**：性能略损（多一跳消息），单机延迟可忽略；代码层面在 `Twin` 基类强制约束（不暴露内部 memory 引用）。

### D-018 知识空缺 escalation 阈值：RAG top-1 score < 0.6

**状态**：已决
**日期**：2026-04-28
**关联**：[A2A协议草案.md §8.1](./A2A协议草案.md)

**背景**：
分身在 RAG 召回结果不足以支撑判断时应主动 escalate，需要量化"知识空缺"。

**决策**：当 RAG 召回 top-1 的最终 score（cosine × weight）< 0.6 时，触发 `reason=knowledge_gap` 的 escalation。

**理由**：0.6 是经验阈值，足够过滤明显无关的召回；真实数据 tune 后可能调整，先给个起点。

**后果与权衡**：阈值过低会漏 escalate，过高会过度打扰本人；比赛期收集召回分布数据后可重新校准。

### D-019 置信度 escalation 阈值：confidence < 0.5

**状态**：已决
**日期**：2026-04-28
**关联**：[A2A协议草案.md §8.1](./A2A协议草案.md)

**背景**：
分身对自身判断的置信度低时应主动 escalate。

**决策**：分身在生成响应时同时输出 `metadata.confidence`，<0.5 时不直接回复，转 escalate（reason=low_confidence）。

**理由**：50% 是"真假难辨"的阈值，低于此值的判断不应在分身间通讯传播；通过 prompt 让 LLM 自评，初期可用。

**后果与权衡**：LLM 自评 confidence 的可靠性需验证；阈值参数化，比赛期可调。
