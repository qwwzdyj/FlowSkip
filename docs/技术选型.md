## 三、各层详细选型

### 3.1 接入层（飞书）

| 项 | 选择 | 备选 / 放弃 | 理由 |
|---|---|---|---|
| SDK | **`lark-oapi` 官方 Python SDK** | 自己封 REST | 全 API 覆盖；事件回调有现成框架 |
| 事件接收方式 | **WebSocket 长连接** | HTTP 回调 | 免公网部署、本地开发即可联调；MVP 阶段不需要负载均衡 |
| 应用形态 | **企业自建应用 + 机器人** | 商店应用 | 比赛期不上架；自建应用权限灵活 |
| 身份呈现 | **应用代发 + 名字带"·分身"后缀**（呼应 D-002） | 独立机器人账号 | 头像复用本人，避免账号洪水；标识由消息卡片 footer 二次强化 |

**需要申请的飞书权限范围（粗清单）**：
- `im:message`（读写消息）
- `im:chat`（群信息）
- `bitable:app`（多维表格读写）
- `docx:document`（文档读写）
- `contact:user.id:readonly`（解析@对象）
- `approval:instance`（后续如需触发审批）

> 实际申请清单留给第三阶段实操，本文只占位。

### 3.2 Agent 运行时（Claude）

| 项 | 选择 | 理由 |
|---|---|---|
| Agent 框架 | **Claude Agent SDK (Python)** | 一等公民；自带 memory / tool use / streaming |
| 主推理模型 | **Claude Opus 4.x** | 分身决策、冲突仲裁、escalation 摘要 |
| 轻量模型 | **Claude Haiku 4.x** | 聊天 compact、消息分类、低风险快速回复 |
| Memory | **Claude auto memory + 项目级 memory file** | 分身的"个人上下文"由 auto memory 承担；规则类（授权范围）由 memory file 提供 |
| Tool 注入 | 自定义工具集合：`lark.*` / `rag.*` / `a2a.*` / `escalate.*` | 详见 [MVP骨架.md](./MVP骨架.md) |

**模型路由策略（MVP 简化版）**：
- 解析用户意图、生成 compact、消息分类 → Haiku
- 协同决策、生成给本人的 escalation 摘要、分身间冲突调和 → Opus

### 3.3 A2A 通讯传输（→ 待登记 D-007）

A2A 协议本身在 [A2A协议草案.md](./A2A协议草案.md) 定义，本节只决定**传输方式**。

| 方案 | 优点 | 缺点 |
|---|---|---|
| **进程内事件总线（asyncio queue）** ✅ MVP | 零基础设施；端到端可单机跑通；开发调试快 | 不能跨机扩展 |
| Redis pub/sub | 跨进程；可解耦 | 多一个依赖；MVP 用不到 |
| 飞书消息当总线 | "分身互发消息"语义直观 | 增加延迟；受飞书 rate limit；调试困难 |
| HTTP/gRPC 互调 | 服务边界清晰 | 对 MVP 过重 |

**决策**：MVP 用进程内事件总线，但**对外抽象成 `MessageBus` 接口**，后续可替换 Redis/NATS。

### 3.4 知识层

#### 3.4.1 Embedding 模型（→ 待登记 D-008）

| 方案 | 优点 | 缺点 |
|---|---|---|
| **BGE-large-zh-v1.5** ✅ | 开源；中文 SOTA 之一；本地部署免费；和 reranker 同家族搭配好 | 需要本地 GPU 或 CPU 推理（768 维向量 CPU 也能扛） |
| bce-embedding-base_v1（网易） | 中英双语强；接入简单 | 社区生态略小 |
| OpenAI text-embedding-3-large | 质量稳定 | 走外网；成本可控但持续付费；中文不如 BGE |
| 阿里 DashScope embedding | 国内合规、中文好 | 锁云厂商 |

选 BGE 的关键理由：**比赛 demo 环境要可控**（不依赖外网），且和 `bge-reranker` 系列天然组合。

#### 3.4.2 Reranker（可选，第三阶段评估）

倾向 `bge-reranker-base`，仅在召回 top-K 后做二阶段重排。MVP 第一版可不上，先用纯加权向量召回看效果。

#### 3.4.3 加权策略（呼应 §四.2 项目概要）

`final_score = cosine(q, doc) × weight(doc)`，其中：

```
weight(doc) = recency_factor(doc) × frequency_factor(doc)

recency_factor(doc) = exp(-λ × age_in_days)         # λ 默认 0.05
frequency_factor(doc) = 1 + log(1 + hit_count)       # 对数衰减避免热点垄断
```

具体落到一条 SQL（pgvector 风格）：

```sql
SELECT id, content, embedding <=> :q AS distance, weight
FROM knowledge_items
ORDER BY (embedding <=> :q) / weight ASC  -- distance / weight：weight 越大排得越前
LIMIT :k;
```

参数（`λ` / 频次饱和系数）通过配置文件可调，方便比赛期 tune。

#### 3.4.4 分块（chunking）策略

| 数据源 | 分块方式 |
|---|---|
| 群聊消息 | 按"会话窗口"切（连续 30 分钟内同一话题为一块），每块跑 compact 生成事实/决策卡片再入库 |
| 飞书文档 | 按段落 + 标题层级切，长段再按 800 字符滑窗 |
| 多维表格 | 按行切，每行一个知识条目，列名作为字段 |

#### 3.4.5 Compact 引擎

用 Claude Haiku 跑提示词（template 见 [MVP骨架.md](./MVP骨架.md) 中的 `prompts/compact.md`），输入是原始聊天块，输出是结构化 JSON：

```json
{
  "facts": ["..."],
  "decisions": ["..."],
  "todos": [{"owner": "...", "task": "..."}],
  "people": ["..."],
  "topic": "..."
}
```

每个 fact / decision 单独入库为一个 `knowledge_item`。

### 3.5 数据存储（PostgreSQL）

| 表 | 用途 |
|---|---|
| `knowledge_items` | 向量 + content + metadata + weight 字段（`created_at` / `last_hit_at` / `hit_count`） |
| `twin_state` | 每个分身的 owner、授权范围、人格化配置 |
| `a2a_messages` | A2A 通讯日志（审计 + 可解释性） |
| `escalations` | 升级队列：原始上下文、建议选项、本人回流结果 |
| `chat_sources` | 抓取源配置（群白名单 / 私聊 opt-in 列表） |
| `audit_logs` | 所有分身行为的可追溯日志（呼应 §八.2 可解释性） |

**扩展**：
- `pgvector`（必须）
- `pg_trgm`（中文模糊匹配兜底，备用）

**ORM 选择**：**SQLAlchemy 2.0（async 模式）+ Alembic 迁移**。理由：稳定、生态成熟、async 支持完善。备选 SQLModel 但和 pgvector 集成略麻烦。

### 3.6 应用框架与调度

| 项 | 选择 | 理由 |
|---|---|---|
| Web 框架 | **FastAPI** | async-first；自动 OpenAPI；与 pydantic 协同 |
| ASGI server | **Uvicorn** | FastAPI 标配 |
| 定时任务 | **APScheduler**（async 模式） | 纯 Python、无需 broker；MVP 阶段够用；ETL 和 compact 都是低频任务 |
| 异步 HTTP 客户端 | **httpx** | async 支持好，飞书 SDK 之外的外部调用统一走它 |

**为什么不上 Celery / arq**：MVP 单机部署，APScheduler 在同进程跑足够；引入 broker 的运维成本不划算。后续若要拆服务，再换。

### 3.7 工程基础

| 项 | 选择 | 备注 |
|---|---|---|
| 包管理 | **uv** | 快、原生 pyproject.toml、`uv lock` 比 pip-tools 友好 |
| 配置管理 | **pydantic-settings + .env** | 类型安全；区分本地/比赛环境 |
| 日志 | **structlog**（结构化 JSON 输出） | 配合 trace_id 做 A2A 链路追踪 |
| 格式化 / Lint | **ruff**（替代 black + isort + flake8） | 一个工具搞定；速度极快 |
| 测试 | **pytest + pytest-asyncio + pytest-postgresql** | postgres fixture 用真实 PG 测加权 SQL |
| 部署 | **Docker Compose（postgres + app）** | 比赛 demo 一键起；不引入 k8s |
| Secrets | `.env` + 飞书 app_secret / Anthropic key 环境变量 | 不进 git |

## 六、暂不在 MVP 选型范围内

刻意排除的项，避免范围蔓延：

- **OpenTelemetry / 全链路追踪**：MVP 用 structlog + trace_id 自给自足
- **k8s / 多副本部署**：单机 docker-compose 即可
- **前端管理控制台**：管理用 FastAPI 自动生成的 Swagger UI 临时顶替
- **Reranker**：第一版走纯加权召回，效果不够再加 BGE-reranker
- **多租户隔离**：比赛是单组织 demo，不做 tenant 拆分
- **审计日志的不可篡改性**：`audit_logs` 表纯追加即可，不上 hash chain / 区块链
